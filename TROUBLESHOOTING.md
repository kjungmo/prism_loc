# Troubleshooting

Common `prism_loc` failure modes, in symptom → cause → fix form. If you're not
sure where to start, run through the ["Verify it's working"](README.md#verify-its-working)
steps in the README first — they narrow down whether the problem is pose/TF,
RViz, or a specific sensor.

| Symptom | Likely cause | Fix |
|---|---|---|
| Node starts and logs `"prism_loc up: backend=..."` / `"prism_loc_fusion (fusion3d) up"`, but `~/pose` never publishes and `map→odom` TF never appears | `laser2d`/`ndt3d`: the particle filter is never seeded — no `/initialpose` was published, `set_initial_pose` is `false`, and `try_global_localization` is also `false` (the default), so nothing initializes the filter | Publish `/initialpose` (RViz "2D Pose Estimate"), or set `set_initial_pose:=true` with `initial_pose_x/y/yaw`, or set `try_global_localization:=true` to auto-recover pose from the first scan via branch-and-bound |
| Same as above, but for **`fusion3d`** | `fusion3d` self-initializes from the first valid GNSS fix + IMU attitude; if GNSS never validates (see the GNSS row below) and no `/initialpose` was sent, `tryInitialize()` never fires | Fix the GNSS thresholds (below) or publish `/initialpose` to seed manually |
| `ndt3d`/`fusion3d`: node exits immediately with a fatal error after logging `"ndt3d: map_pcd_path is empty"`, `"ndt3d: failed to load PCD or empty: ..."`, or the `fusion3d:` equivalents | `map_pcd_path` is empty, points at a nonexistent file, or the `.pcd` failed to parse/is empty — the map is a mandatory input, so the node fails fast instead of running without it | Pass an absolute, existing `.pcd` path via `map_pcd_path:=/abs/path/to/map.pcd`; verify it loads with `pcl_viewer /abs/path/to/map.pcd` |
| Everything above looks fine (filter is seeded, map loaded) but pose/TF still never publish, or only publish once and then stop | `use_sim_time:=true` was set but nothing is publishing `/clock` (no bag playing with `--clock`, no simulator) — ROS time stalls at 0 and stamped lookups/timers never advance | Leave `use_sim_time` at its default `false` for a real robot; if replaying a bag, play it with `ros2 bag play <bag> --clock` so `/clock` is published |
| RViz opens but the view is blank (no map, no particles, no pose) | Fixed Frame in RViz isn't set to `map` (or whatever `global_frame` is configured to), or the map/particle-cloud displays aren't subscribed to the right topics | Set RViz's Fixed Frame to `map`; add/point the `Map` display at `/map` (laser2d) and the pose/particle-cloud displays at `~/pose` / `~/particle_cloud` under the node's private namespace (e.g. `/prism_loc/particle_cloud`) |
| Log repeats `"odom TF: ..."` (throttled warning) | Missing or broken `odom_frame → base_frame` TF (default `odom → base_link`) — nothing is publishing wheel/base odometry as a TF | Make sure your odometry source publishes `odom → base_link` (e.g. a diff-drive controller or `robot_localization`); check with `ros2 run tf2_ros tf2_echo odom base_link` |
| Log repeats `"sensor TF: ..."` (throttled warning), or `fusion3d` logs `"fusion3d: no odom->base_link; publishing map->base_link"` | Missing or broken `base_frame → sensor_frame` static TF (e.g. `base_link → laser`/`base_link → lidar`) — the sensor extrinsic was never published | Add a `static_transform_publisher` (or robot's URDF `robot_state_publisher`) that publishes `base_link → <your sensor frame>`; the sensor frame must match the `frame_id` in the incoming scan/cloud/imu message header |
| No pose/TF, and the node logs a warning every 10 s like `"no messages on /scan (0 publishers) - check scan_topic and the sensor driver"` (or the equivalent for `points_topic`/`map_topic`/`imu_topic`/`gnss_topic`) | A topic-name typo/mismatch: `scan_topic`, `points_topic`, `map_topic`, `imu_topic`, or `gnss_topic` doesn't match what your driver actually publishes, so the startup watchdog keeps flagging the required input it has never received. A publisher count of `0` confirms nothing is publishing on the name the node subscribed to | Read the topic named in the warning and the publisher count: `0` means the name is wrong. Check the defaults (`/scan`, `/points`, `/map`, `/imu`, `/gnss`) against `ros2 topic list`; override with the matching launch/param, e.g. `scan_topic:=/my_lidar/scan`, or remap the driver instead |
| `fusion3d`: `/gnss` messages are arriving (`ros2 topic hz /gnss` is nonzero) but the filter never uses them / never initializes position from GNSS | `msg.status.status < gnss_min_status` (fix not good enough) or `msg.position_covariance[0] > gnss_max_pos_cov` (fix too noisy) is silently dropping every message in `onGnss()` | Check your receiver's actual `NavSatStatus.status` and `position_covariance[0]` values (`ros2 topic echo /gnss`); relax `gnss_min_status` (default `0`) / raise `gnss_max_pos_cov` (default `25.0`) to match your receiver, or fix the receiver's fix quality |
| A topic looks like it should have data (`ros2 topic list` shows it, a publisher exists) but `ros2 topic echo` shows nothing and `prism_loc` never reacts | QoS mismatch between publisher and subscriber. `/map` is subscribed with **transient_local + reliable** (durable/"latched", matching `nav2_map_server`'s default) — a plain volatile map publisher will never be seen. `/scan`, `/points`, `/imu`, `/gnss` are subscribed with **SensorDataQoS** (best-effort) — a publisher that strictly requires reliable delivery can still be incompatible depending on its profile | Compare profiles with `ros2 topic info -v <topic>`; make sure a custom `/map` publisher uses transient_local+reliable, and that sensor drivers use a QoS compatible with `SensorDataQoS` (best-effort, small history depth) |

See also [`PARAMS.md`](PARAMS.md) for the full list of parameters referenced
above, and [`SPEC.md`](SPEC.md) for the TF/topic contract each backend
implements.
